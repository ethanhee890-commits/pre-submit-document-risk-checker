# 외부 데모 샘플 문서 세트

## 1. 목적

제한적 외부 데모는 실제 사용자 문서가 아니라 사전 제작된 샘플 문서로만 진행한다.

목표는 분석 정확도 검증이 아니라 화면 흐름, 리포트 이해도, PDF 가치, 문서함/설정 정책 이해도를 확인하는 것이다.

## 2. 데모 고지

데모 시작 전 아래 문구를 읽거나 화면에 표시한다.

```text
이 화면은 제품 흐름 검증을 위한 프로토타입 데모입니다.
현재 결과는 샘플 문서 기준으로 표시됩니다.
실제 제출 판단이나 작성자 판정의 근거로 사용하지 마세요.
```

## 3. 샘플 문서 원칙

| 원칙 | 설명 |
|---|---|
| 실제 개인정보 없음 | 이름, 전화번호, 이메일, 주소 제외 |
| 실제 제출 문서 아님 | 외부 제출 가능한 실제 문서 사용 금지 |
| 허위 수치 최소화 | 수치는 샘플임을 표시하거나 일반 예시로 사용 |
| 민감 문서 금지 | 채용, 평가, 과제, 내부 보고 원문 사용 금지 |
| 문서 유형별 다양성 | 자소서, 리포트, 블로그, 업무문서 각 2개 이상 |
| 양호 샘플 포함 | 모든 샘플이 문제처럼 보이지 않도록 낮은 리스크 샘플 포함 |

## 5. DEMO-RESUME-01: 자소서 보완 필요

```text
저는 성장 과정에서 여러 경험을 통해 책임감과 문제 해결력을 키워 왔습니다.
특히 팀 활동을 하면서 다양한 의견을 조율했고, 맡은 일을 끝까지 해내는 태도를 배웠습니다.
앞으로도 이러한 장점을 바탕으로 조직에 기여하는 사람이 되고 싶습니다.
```

기대 결과:

| 항목 | 기대 |
|---|---|
| 주요 finding | 경험의 배경, 본인 역할, 결과가 부족함 |
| 개선 제안 | 실제 경험명, 맡은 역할, 결과 수치를 넣는 문장 구조 |
| 리스크 축 | 문체 점검 보완 권장, 제출 안정성 우선 확인 |

## 6. DEMO-RESUME-02: 자소서 양호

```text

기존 안내문은 일정과 준비물이 흩어져 있어 질문이 반복되었고, 저는 신청 절차, 첫 모임 준비물, 담당자 연락 방법을 한 장으로 묶었습니다.
그 결과 첫 모임 전 반복 문의가 줄었고, 이후 행사 안내문도 같은 형식으로 정리했습니다.
```

기대 결과:

| 항목 | 기대 |
|---|---|
| 주요 finding | 큰 위험보다 문장 길이 조정 정도 |
| 개선 제안 | 결과를 더 구체화할 수 있는 선택 제안 |
| 리스크 축 | 전반적으로 양호 또는 일부 보완 권장 |

## 8. DEMO-REPORT-02: 리포트 양호

## 13. 1분 데모 스크립트

```text
이 데모는 실제 문서 제출 여부를 판단하는 서비스가 아니라, 제출 전에 문장과 근거를 스스로 확인할 수 있게 돕는 프로토타입입니다.
오늘은 실제 개인 문서가 아니라 사전 제작된 샘플 문서만 사용합니다.
먼저 샘플 문서를 선택하고, 4축 리포트에서 어떤 부분을 먼저 확인하면 좋을지 봅니다.
그다음 문장별 근거와 개선 제안을 비교해 제안이 실제 수정에 도움이 되는지 확인합니다.
마지막으로 PDF와 문서함 안내가 충분히 이해되는지 의견을 받겠습니다.
```

## 14. 설명 금지 기준

- 실제 사용자 원문 붙여넣기 유도
- 분석 정확도 검증으로 설명
- 작성자 판정 도구처럼 소개
- 민감한 자소서, 과제, 업무문서 입력
- 결과를 제출 가능 여부의 근거로 안내
- 특정 문장이 문제라고 단정하는 표현

## 15. Q&A 가이드

| 질문 | 답변 방향 |
|---|---|
| 실제 제 문서를 넣어도 되나요? | 이번 외부 데모에서는 샘플 문서만 사용한다고 안내 |
| 결과가 정확한가요? | 제출 전 참고 신호이며 최종 판단은 사용자가 직접 확인한다고 안내 |
| 점수가 낮으면 불이익이 있나요? | 불이익 판단용이 아니며 보완 우선순위를 정하는 용도라고 안내 |
| PDF를 다른 사람에게 보내도 되나요? | 샘플 PDF만 공유 가능하며 실제 문서 PDF는 민감정보 확인 후 사용해야 한다고 안내 |

## 16. 피드백 질문

| 질문 | 목적 |
|---|---|
| 4축 리포트가 단일 점수보다 이해하기 쉬웠나요? | 리포트 구조 검증 |
| 문장별 근거가 충분히 구체적이었나요? | 근거 UX 검증 |
| 개선 제안이 실제 수정에 도움이 될 것 같나요? | 제안문 가치 검증 |
| PDF 리포트를 저장하거나 공유할 이유가 있나요? | PDF 가치 검증 |
| 데이터 보관 안내가 충분히 명확했나요? | 정책 이해도 검증 |

피드백 수집은 설문 답변, 관찰 메모, 화면 흐름에 대한 의견으로 제한한다. 실제 원문, 개인 연락처, 제출 예정 문서는 수집하지 않는다.
