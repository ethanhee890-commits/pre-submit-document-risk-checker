# Analysis v0 수용 기준

## 1. 목적

이 문서는 rule-based analysis v0 구현 전 승인 기준을 정의한다. 구현은 이 문서가 승인된 뒤 제한적으로 진행한다.

Analysis v0는 작성자나 작성 도구를 판정하지 않고, 제출 전 사용자가 직접 확인할 수 있는 문체·출처·제출 안정성 신호를 정리하는 규칙 기반 엔진이다.

## 2. 구현 허용 범위

| 항목 | 허용 |
|---|---|
| 분석 방식 | 클라이언트 또는 stateless 서버의 규칙 기반 분석 |
| 입력 처리 | 분석 중 일시 처리, 기본 미보관 |
| 출력 | 4축 리포트, 문장별 근거, 개선 제안, 체크리스트 |
| 저장 | 현재 브라우저 localStorage 저장만 유지 |
| PDF | 기존 브라우저 저장 우선, 서버 PDF는 QA/제한 기능 유지 |

## 3. 구현 금지 범위

- 실제 AI API 호출
- 사용자 원문 서버 저장
- 계정 기반 문서함
- B2B 조직 저장/권한 시스템
- 공유 링크
- 외부 데모 입력 수집 기능
- 결과를 작성자 판정처럼 표현하는 UI

## 4. 입력 전처리 기준

| 처리 | 기준 |
|---|---|
| 공백 정리 | 연속 공백과 빈 줄을 표시용으로 정리하되 원문 의미는 바꾸지 않음 |
| 글자 수 | 공백 포함/제외 중 하나를 선택해 일관되게 계산, UI에는 공백 포함 기준 권장 |
| 문장 분리 | `.`, `?`, `!`, `다.`, `요.`, 줄바꿈을 우선 기준으로 분리 |
| 최소 입력 | 빈 문서 또는 20자 미만은 분석하지 않고 입력 오류 표시 |
| 긴 입력 | 10,000자 초과 시 입력 제한 또는 잘림 없는 안내 표시 |

## 5. 신뢰도 수용 기준

| 조건 | 결과 |
|---|---|
| 200자 미만 또는 3문장 미만 | `analysisConfidence=low`, 신뢰도 안내 필수 |
| 200-499자 또는 3-5문장 | `analysisConfidence=medium`, 신뢰도 안내 권장 |
| 500자 이상이고 6문장 이상 | `analysisConfidence=high`, 일반 리포트 표시 |

신뢰도는 리스크 등급을 대신하지 않는다. 짧은 문서에서 `우선 확인`이 표시되더라도 "참고 신호가 제한적"이라는 안내를 함께 보여야 한다.

## 6. 축별 scoring 기준

내부 점수는 0-100으로 계산하되 UI에는 직접 노출하지 않는다.

### 6.1 문체 점검

| 신호 | 점수 |
|---|---:|
| 같은 어미 또는 문장 구조 3회 이상 반복 | +15 |
| 추상 평가어 3개 이상 | +15 |
| 한 문장 120자 초과 | +10 |
| "매우", "다양한", "좋은", "훌륭한" 등 범용 수식어 반복 | +10 |
| 구체 행동·대상·결과가 함께 있는 문장 2개 이상 | -15 |

### 6.2 AI 작성 리스크

이 축은 실제 작성자나 작성 도구 판정이 아니라 작성 패턴 점검 신호다.

| 신호 | 점수 |
|---|---:|
| 문장 길이 편차가 낮고 구조가 과도하게 균일함 | +15 |
| 일반론 중심 문장 3개 이상 | +15 |
| 문단 간 구체 맥락 없이 매끄럽게만 연결됨 | +10 |
| 본인 경험, 수치, 고유 맥락이 거의 없음 | +15 |
| 원인-행동-결과가 구체적으로 연결됨 | -15 |

### 6.3 출처·인용 리스크

| 신호 | 점수 |
|---|---:|
| 수치 또는 퍼센트가 있는데 출처 표현이 없음 | +25 |
| "대부분", "최근", "증가", "감소", "연구" 등 근거 필요 표현 | +15 |
| 조사명·기관명·연도 중 2개 이상 없음 | +15 |
| 주장과 근거가 한 문장 안에서 연결됨 | -10 |
| 출처 슬롯 또는 참고문헌 위치가 있음 | -15 |

### 6.4 제출 안정성

| 신호 | 점수 |
|---|---:|
| 문서 목적이 불명확함 | +20 |
| 자소서에서 역할·행동·결과 중 2개 이상 누락 | +20 |
| 업무문서에서 담당자·기한·산출물 중 2개 이상 누락 | +25 |
| 리포트에서 결론과 근거 연결이 약함 | +15 |
| 제출 전 확인 체크리스트가 모두 충족됨 | -20 |

## 7. 등급 산정 기준

| 내부 점수 | 사용자 표시 |
|---:|---|
| 0-39 | 양호 |
| 40-69 | 보완 권장 |
| 70-100 | 우선 확인 |

점수는 0보다 낮아지지 않고 100을 넘지 않도록 clamp한다.

## 8. 종합 등급 기준

1. 문서 유형별 가중치를 적용해 weighted score를 계산한다.
2. weighted score의 등급을 기본 `overallLevel`로 둔다.
3. critical finding이 있으면 `overallLevel`을 최소 `우선 확인`으로 올린다.
4. 한 축이 `우선 확인`이고 문장 근거가 2개 이상이면 `overallLevel`을 최소 `보완 권장`으로 올린다.
5. `analysisConfidence=low`인 경우 종합 등급 옆에 신뢰도 안내를 필수 표시한다.

## 9. critical finding 정의

| ID | 조건 | 종합 영향 |
|---|---|---|
| CRIT-PII-01 | 고유식별정보 패턴 감지 | `우선 확인` |
| CRIT-CITE-01 | 핵심 결론에 수치·조사 주장이 있으나 출처 없음 | `우선 확인` |
| CRIT-BIZ-01 | 업무문서에서 담당자와 기한이 모두 없음 | `우선 확인` |
| CRIT-RESUME-01 | 자소서 전체가 추상 장점 나열 중심이고 실제 경험 근거 없음 | 최소 `보완 권장` |
| CRIT-LENGTH-01 | 20자 미만 또는 비어 있음 | 분석 중단, 입력 오류 |

## 10. 민감정보 감지 기준

| 범주 | 1차 패턴 | 처리 |
|---|---|---|
| 이메일 | `local@domain` 형식 | 마스킹 안내 |
| 전화번호 | `010-0000-0000`, `02-000-0000` 유사 형식 | 마스킹 안내 |
| 고유식별번호 | `000000-0000000` 유사 형식 | 우선 확인, 저장 금지 권고 |
| 주소 | 시/군/구와 상세 번지 조합 | 상세 주소 마스킹 권고 |
| 학번/사번 | "학번", "사번", "ID" 뒤 숫자열 | 필요 최소 표시 권고 |

오탐 가능성이 있으므로 UI에서는 "포함될 수 있습니다"로 표현하고, 민감정보라고 단정하지 않는다.

## 11. 제안문 템플릿 기준

| 문서 유형 | 제안 원칙 | 예시 |
|---|---|---|
| 자소서 | 실제 경험을 사용자가 넣도록 슬롯 제공 | `[상황]에서 저는 [역할]을 맡아 [행동]했고, 그 결과 [변화]를 확인했습니다.` |
| 리포트 | 허위 출처를 만들지 않고 출처 슬롯 제공 | `[기관/자료명, 연도]에 따르면 [대상]에서 [결과]가 확인되었습니다.` |
| 블로그 | 독자 상황과 실제 사용 장면을 넣음 | `[상황]에서는 [방법]을 먼저 적용하면 [변화]를 확인하기 쉽습니다.` |
| 업무문서 | 담당자, 기한, 산출물을 포함 | `[담당자]가 [기한]까지 [산출물]을 정리해 [공유 위치]에 공유하겠습니다.` |
| 기타 | 문서 목적과 확인 기준을 먼저 묻는 형태 | `이 문서는 [목적]을 위해 작성되었으며, 제출 전 [확인 기준]을 먼저 점검하겠습니다.` |

제안문은 사용자가 검토 없이 그대로 제출하도록 유도하지 않는다. 없는 경험, 수치, 출처는 생성하지 않는다.

## 12. 테스트 입력 세트

| 테스트 ID | 문서 유형 | 입력 조건 | 기대 결과 |
|---|---|---|---|
| AV0-EMPTY-001 | 기타 | 빈 입력 | 분석 중단, 오류 메시지 |
| AV0-SHORT-001 | 기타 | 200자 미만 | 신뢰도 low, 짧은 문서 안내 |
| AV0-RESUME-001 | 자소서 | 추상 장점 반복 | 문체/제출 안정성 보완 권장 이상 |
| AV0-RESUME-002 | 자소서 | 구체 경험·역할·결과 포함 | 전반 양호 또는 일부 보완 |
| AV0-REPORT-001 | 리포트 | 수치·대부분 표현, 출처 없음 | 출처·인용 우선 확인 |
| AV0-REPORT-002 | 리포트 | 출처 슬롯과 일반화 제한 포함 | 출처·인용 양호 또는 보완 권장 |
| AV0-BLOG-001 | 블로그 | 범용 평가어 반복 | 문체 우선 확인 |
| AV0-BIZ-001 | 업무문서 | 담당자·기한·산출물 누락 | 제출 안정성 우선 확인 |
| AV0-PII-001 | 기타 | 이메일·전화번호 포함 | 민감정보 안내와 마스킹 제안 |
| AV0-LONG-001 | 기타 | 10,000자 초과 | 입력 제한 안내 |

## 13. UI 수용 기준

| 화면 | 기준 |
|---|---|
| `/check` | 입력 오류, 짧은 문서 안내, 문서 유형 선택이 동작 |
| `/checking` | 분석 중 상태가 1초 이상 표시되어 흐름 인지 가능 |
| `/report` | 4축 등급, 종합 등급, 신뢰도 안내, 문장별 근거 표시 |
| 문장별 분석 | finding 클릭 시 원문, 이유, 제안문 확인 가능 |
| 개선 제안 | 지시문 단독이 아니라 비교 가능한 제안 또는 슬롯형 문장 제공 |
| 체크리스트 | 현재 화면에서 체크 상태 변경 가능 |
| PDF | 리포트 요약과 제한된 근거 문장만 포함 |

## 14. QA 통과 기준

| 항목 | 통과 기준 |
|---|---|
| 단위 테스트 | 테스트 입력 세트 전부 기대 결과와 일치 |
| 금지 표현 검색 | 회피성·판정성 표현 미검출 |
| 데이터 정책 | 원문 서버 저장 없음, 로그에 원문 없음 |
| 반응형 | 모바일 390px, 데스크톱 1440px에서 주요 흐름 확인 |
| 회귀 | 기존 smoke test 통과 |
| 배포 | `/api/version`이 승인 커밋 전체 hash 반환 |

## 15. 승인 조건

Analysis v0 구현 착수 전 아래가 모두 확인되어야 한다.

- 에던이 본 문서 승인
- 제니가 본 문서 승인 또는 보완 사항 없음 판정
- 데이터 정책 copy spec과 충돌 없음
- 외부 데모 문서와 샘플 expected report가 충돌 없음
- 배포본 commit과 GitHub commit 일치
