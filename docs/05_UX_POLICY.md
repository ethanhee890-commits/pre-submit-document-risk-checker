# UX 정책 v0.1

## 1. 핵심 UX 원칙

| 원칙 | 설명 |
|---|---|
| 판정이 아니라 점검 | 작성자를 단정하지 않고 검토가 필요한 신호를 보여준다. |
| 점수보다 근거 | 단일 점수보다 문장별 근거와 개선 방향을 제공한다. |
| 불안 조장 금지 | 사용자를 겁주기보다 다음 행동을 안내한다. |
| 회피 기능 금지 | 탐지기 통과율 개선, AI 점수 낮추기 기능을 제공하지 않는다. |
| 한국어 가독성 | 긴 한글 문장, 버튼 폭, 행간, 줄바꿈을 고려한다. |

## 2. 결과 표현 정책

| 항목 | 정책 |
|---|---|
| 등급 | 낮음/보통/높음 3단계 사용 |
| 퍼센트 | 필요 시 내부값으로만 사용, 전면 노출 지양 |
| 색상 | 빨강 과다 사용 금지, 주의/검토/안정 톤 |
| 문구 | “가능성”, “리스크”, “추가 검토” 중심 |
| 면책 | 결과는 참고 지표이며 단독 판단 근거가 아님을 표시 |

## 3. 금지 문구

아래 표현은 UI, 랜딩, 버튼, 리포트, 토스트, README에서 사용하지 않는다.

- AI 탐지 우회
- 탐지기 통과
- 통과 보장
- Turnitin 회피
- 100% 사람 글
- AI 냄새 제거
- undetectable
- bypass
- humanizer
- detector proof
- AI 점수 낮추기
- 절대 판정
- 부정행위 방지 확정 판정

## 4. 권장 문구

| 상황 | 권장 문구 |
|---|---|
| 제품 설명 | 제출 전 문서 리스크 점검 |
| 결과 제목 | 문서 신뢰도 리포트 |
| AI 관련 | AI 작성 리스크 |
| 위험 문장 | 추가 검토가 필요한 문장 |
| 결과 안내 | 작성자를 단정하지 않는 참고 지표 |
| 개선 기능 | 문체와 명확성 개선 제안 |
| 출처 문제 | 출처·인용 보완 필요 |
| 짧은 글 | 분석 신뢰도가 낮을 수 있습니다. |

## 5. 사용자 감정 설계

| 상황 | 피해야 할 표현 | 권장 표현 |
|---|---|---|
| 고위험 | “AI가 쓴 글입니다” | “일부 구간에서 추가 검토가 필요한 신호가 발견되었습니다.” |
| 출처 부족 | “표절 위험” 단정 | “출처 또는 인용 보완이 필요한 주장으로 보입니다.” |
| 문체 문제 | “AI 냄새가 납니다” | “문장 구조가 반복적이고 지나치게 정돈되어 보일 수 있습니다.” |
| 낮은 리스크 | “안전합니다” | “현재 기준에서 큰 위험 신호는 낮습니다.” |
| 짧은 글 | “분석 불가” | “짧은 문서는 분석 신뢰도가 낮을 수 있습니다.” |

## 6. 버튼 정책

| 버튼 | 문구 |
|---|---|
| Primary CTA | 문서 점검 시작하기 |
| 검사 실행 | 리스크 점검하기 |
| 결과 저장 | 리포트 저장 |
| 다운로드 | 리포트 다운로드 |
| 재검사 | 다시 점검하기 |
| 복사 | 제안문 복사 |

## 7. 상태 정책

| 상태 | UX |
|---|---|
| Empty | 입력 예시와 최소 글자 안내 |
| Loading | 분석 단계별 문구 표시 |
| Error | 원인과 다음 행동 제시 |
| Success | 결과와 우선 확인 항목 제시 |
| Warning | 과도한 경고 대신 보완 방법 안내 |

## 8. 리포트 상단 고정 안내

```text
이 결과는 작성자를 단정하지 않습니다.
문체와 문서 패턴을 바탕으로 제출 전 검토가 필요한 신호를 보여주는 참고 지표입니다.
최종 판단이나 불이익의 단독 근거로 사용해서는 안 됩니다.
```
