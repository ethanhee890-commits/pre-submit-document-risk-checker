# 외부 데모 스크립트와 운영 체크리스트

## 1. 목적

이 문서는 제한적 외부 데모를 샘플 문서 기반으로만 진행하기 위한 운영 스크립트다.

외부 데모는 제품 흐름, 리포트 이해도, PDF 가치, 데이터 정책 이해도를 확인하는 자리이며 실제 사용자 문서를 분석하거나 수집하지 않는다.

## 2. 진행 조건

| 조건 | 기준 |
|---|---|
| 배포 버전 | `/api/version`이 승인 커밋 전체 hash를 반환 |
| 샘플 문서 | `docs/24`의 샘플 ID만 사용 |
| 데이터 입력 | 실제 원문, 개인정보, 제출 예정 문서 입력 금지 |
| 피드백 | 화면 이해도, 문구, 흐름, 신뢰도 의견만 수집 |
| PDF | 샘플 문서 기준 PDF만 저장 또는 공유 |
| 진행자 | 프로토타입 고지와 금지 설명을 먼저 읽음 |

## 3. 데모 전 체크리스트

| 체크 | 기준 |
|---|---|
| 배포 URL 접속 | `https://pre-submit-document-risk-checker.onrender.com` 정상 접속 |
| 버전 확인 | `/api/version` commit이 승인 commit과 일치 |
| 금지 표현 검색 | 사용자 화면에 회피성·보장성 표현 없음 |
| 샘플 문서 준비 | DEMO-RESUME/REPORT/BLOG/BIZ 샘플 준비 |
| 피드백 폼 준비 | 실제 원문/개인정보 입력 금지 문구 포함 |
| PDF 저장 확인 | 샘플 리포트 PDF 저장 가능 |

## 4. 1분 소개 스크립트

```text
오늘 보실 화면은 제출 전 문서를 스스로 점검하는 흐름을 확인하기 위한 프로토타입입니다.
실제 제출 여부나 작성자를 판단하는 도구가 아닙니다.

데모에서는 실제 개인 문서가 아니라 미리 만든 샘플 문서만 사용합니다.
결과는 문체, AI 작성 리스크, 출처·인용, 제출 안정성 네 가지 축으로 표시됩니다.

리포트의 문장별 근거와 개선 제안이 실제 수정 판단에 도움이 되는지,
PDF와 문서함 안내가 충분히 이해되는지 중심으로 의견을 부탁드립니다.
```

## 5. 표준 진행 순서

```text
랜딩 확인
→ 프로토타입 고지 설명
→ 샘플 문서 선택 또는 붙여넣기
→ 문서 유형 선택
→ 검사 실행
→ 4축 리포트 확인
→ 문장별 근거 확인
→ 개선 제안 비교
→ 체크리스트 확인
→ PDF 저장 흐름 확인
→ 문서함/설정 정책 설명
→ 피드백 수집
```

## 6. 샘플 선택 방식

현재 배포본에 별도 샘플 선택 UI가 없는 경우 진행자가 `docs/24`의 샘플 본문을 붙여넣는다. 향후 데모 전용 샘플 선택 UI를 만들 경우에도 실제 입력 수집으로 보이지 않게 "샘플 문서 선택"이라고 표시한다.

권장 순서:

1. `DEMO-RESUME-01`로 보완 필요 흐름 설명
2. `DEMO-RESUME-02` 또는 `DEMO-BIZ-02`로 양호 샘플 확인
3. `DEMO-REPORT-01`로 출처·인용 흐름 확인
4. 시간이 남으면 PDF 저장과 문서함 안내 확인

## 7. 샘플별 기대 리포트

| 샘플 ID | 문체 | AI 작성 리스크 | 출처·인용 | 제출 안정성 | 기대 finding |
|---|---|---|---|---|---|
| DEMO-RESUME-01 | 보완 권장 | 보완 권장 | 양호 | 우선 확인 | 추상 표현, 실제 경험 근거 부족 |
| DEMO-RESUME-02 | 양호 | 양호 | 양호 | 양호 | 문장 길이 조정 정도 |
| DEMO-REPORT-01 | 보완 권장 | 양호 | 우선 확인 | 보완 권장 | 출처 없는 일반화 주장 |
| DEMO-REPORT-02 | 양호 | 양호 | 보완 권장 | 양호 | 실제 조사명/표본 보강 제안 |
| DEMO-BLOG-01 | 우선 확인 | 보완 권장 | 양호 | 양호 | 범용 평가어 반복 |
| DEMO-BLOG-02 | 양호 | 양호 | 양호 | 양호 | 독자 맥락이 분명함 |
| DEMO-BIZ-01 | 보완 권장 | 양호 | 보완 권장 | 우선 확인 | 담당자·기한·산출물 누락 |
| DEMO-BIZ-02 | 양호 | 양호 | 양호 | 양호 | 실행 항목이 분명함 |

## 8. 샘플별 기대 제안

| 샘플 ID | 기대 제안 방향 |
|---|---|
| DEMO-RESUME-01 | `[경험명]에서 [역할]을 맡아 [행동]했고, [결과]를 확인했습니다.` |
| DEMO-RESUME-02 | 결과 수치나 전후 변화가 있으면 선택적으로 보강 |
| DEMO-REPORT-01 | `[조사명/연도]에 따르면 [대상]에서 [결과]가 확인되었습니다.` |
| DEMO-REPORT-02 | 예시 자료임을 유지하고 실제 조사명·표본 수를 확인 |
| DEMO-BLOG-01 | 독자 상황, 사용 장면, 전후 변화를 넣어 구체화 |
| DEMO-BLOG-02 | 긴 문장을 나누고 핵심 행동을 먼저 배치 |
| DEMO-BIZ-01 | `[담당자]가 [기한]까지 [산출물]을 [공유 위치]에 공유` |
| DEMO-BIZ-02 | 시스템 링크, 담당자 이름, 회의 안건을 선택 보강 |

## 9. 진행 중 금지 설명

- 실제 문서를 넣어도 된다고 말하지 않는다.
- 결과가 정확도 검증을 통과했다고 말하지 않는다.
- 작성자나 작성 도구를 판단한다고 설명하지 않는다.
- 점검 결과가 제출 결과를 바꾼다고 말하지 않는다.
- 기관용 화면을 평가 자동화 도구처럼 설명하지 않는다.
- 데모 피드백 폼에 원문을 붙여넣게 하지 않는다.

## 10. Q&A 스크립트

| 질문 | 답변 |
|---|---|
| 제 문서를 넣어 봐도 되나요? | 이번 데모에서는 샘플 문서만 사용합니다. 실제 문서는 수집하지 않습니다. |
| 결과가 정확한가요? | 제출 전 참고 신호를 정리하는 흐름을 보는 단계입니다. 최종 판단은 사용자가 직접 확인해야 합니다. |
| 낮은 등급이면 문제가 있나요? | 불이익 판단이 아니라 먼저 확인할 항목을 정리한 표시입니다. |
| PDF를 공유해도 되나요? | 오늘은 샘플 PDF만 공유 가능합니다. 실제 문서는 민감정보 확인이 필요합니다. |
| 기관용으로 쓰면 관리자가 원문을 보나요? | 현재 데모에는 실제 원문 열람이나 조직 저장 기능이 없습니다. |

## 11. 피드백 수집 폼

폼 상단 고지:

```text
실제 문서 원문, 이름, 연락처, 제출 예정 문서는 입력하지 마세요.
샘플 문서 데모를 본 뒤 화면 이해도와 개선 의견만 남겨 주세요.
```

수집 가능 항목:

- 4축 리포트 이해도
- 문장별 근거의 구체성
- 개선 제안의 실제 수정 도움 정도
- PDF 저장 필요성
- 데이터 보관 안내 이해도
- 데모 중 헷갈린 문구

수집 금지 항목:

- 실제 문서 원문
- 개인정보
- 제출 예정 문서
- 기관 내부 문서 내용
- 평가 대상자 정보

## 12. 데모 종료 멘트

```text
오늘 확인한 내용은 샘플 문서 기준의 화면 흐름입니다.
실제 사용자 문서 수집이나 제출 판단에는 사용하지 않습니다.

남겨 주신 의견은 리포트 구조, 안내 문구, PDF와 문서함 흐름을 개선하는 데만 참고하겠습니다.
```

## 13. 데모 후 정리

| 항목 | 처리 |
|---|---|
| 브라우저 저장 데이터 | 데모 종료 후 문서함 수동 삭제 |
| PDF 파일 | 샘플 PDF만 보관, 실제 원문 PDF 없음 확인 |
| 피드백 | 원문이나 개인정보가 들어간 답변은 즉시 삭제 |

| 이슈 기록 | 화면 혼선, 문구 개선, PDF 문제만 기록 |

## 14. 외부 데모 승인 조건

제한적 외부 데모 전 아래가 모두 충족되어야 한다.

- 제니가 docs/29 승인
- 에던이 외부 데모 진행 승인
- 배포 URL의 `/api/version`이 승인 commit과 일치
- 데모 진행자가 금지 설명을 숙지
- 피드백 폼에 실제 원문/개인정보 입력 금지 고지 포함
- 데모에서 실제 사용자 문서를 입력하지 않음
