# 개인/B2B 화면 분리 전략

## 1. 목적

개인 MVP와 기관용 데모는 목적, 권한, 데이터 정책, 리포트 해석 방식이 다르다. Sprint 3에서는 두 흐름을 메뉴와 화면 구조에서 명확히 분리한다.

## 2. 분리 원칙

| 원칙 | 설명 |
|---|---|
| 개인 우선 | 기본 공개 화면은 개인 제출 전 점검 흐름 |
| B2B 숨김 | 기관용 화면은 일반 내비게이션에 노출하지 않음 |
| 정책 분리 | 개인 데이터 정책과 조직 데이터 정책을 별도로 표시 |
| 권한 분리 | B2B는 관리자, 검토자, 제출자 역할을 구분 |
| 리포트 목적 분리 | 개인은 자기 점검, B2B는 운영 현황과 정책 관리 |
| 불이익 방지 | 보고서만으로 평가, 징계, 불합격 판단을 하지 않도록 고지 |

## 3. 개인 MVP 정보구조

```text
/
/check
/checking
/report
/pdf-report
/documents
/settings
```

개인 MVP 내비게이션:

- 기능
- 사용 사례
- 안전 원칙
- 문서함
- 설정
- 검사하기

## 4. B2B 후보 정보구조

일반 공개 메뉴에서는 숨긴다.

```text
/org
/org/dashboard
/org/reports
/org/policies
/org/members
/org/audit-logs
```

`/org` 화면 상단에는 고정 고지를 표시한다.

```text
기관용 프로토타입 데모입니다. 샘플 데이터로만 화면 흐름을 확인하세요.
실제 제출자 평가나 원문 보관 기능은 아직 제공하지 않습니다.
```

## 5. 화면 목적 비교

| 화면 | 개인 MVP | B2B |
|---|---|---|
| 랜딩 | 개인 사용자 가치 설명 | 조직용 별도 랜딩 후보 |
| 검사 입력 | 본인 문서 점검 | 조직 정책 기준 점검 |
| 리포트 | 본인 제출 전 보완 | 검토자 코멘트와 정책 확인 |
| 문서함 | 개인 저장 리포트 | 팀/조직 리포트 목록 |
| 설정 | 보관/원문 저장 개인 설정 | 조직 보관기간, 권한, 감사 설정 |
| PDF | 개인 보관용 | 조직 공유/검토용 |

## 6. 권한 모델 초안

| 역할 | 볼 수 있는 데이터 | 원문 접근 | 삭제 권한 | 주요 기능 |
|---|---|---|---|---|
| 제출자 | 본인 리포트, 본인 체크리스트 | 기본 미노출, opt-in 시 본인만 | 본인 저장 데이터 삭제 요청 | 문서 점검, PDF 저장, 보완 체크 |
| 검토자 | 할당된 리포트 요약, 코멘트 | 정책 승인 전 미제공 | 삭제 불가, 관리자 요청 | 코멘트, 체크리스트 확인 |
| 관리자 | 조직 설정, 멤버, 감사 로그 | 정책 승인 전 미제공 | 조직 정책 범위 내 삭제 | 보관기간, 권한, 멤버 관리 |

B2B의 원문 접근은 데이터 정책, 보관/삭제 정책, 법무 검토가 승인되기 전까지 구현하지 않는다.

## 7. 데이터 정책 차이

| 항목 | 개인 MVP | B2B |
|---|---|---|
| 원문 보관 | 기본 미보관 | 관리자 정책 필요 |
| 리포트 보관 | 사용자 선택 | 조직 보관기간 |
| 삭제 | 사용자 직접 삭제 | 사용자/관리자 권한에 따라 삭제 |
| 감사 로그 | 없음 | 필수 후보 |
| 관리자 열람 | 없음 | 역할 기반 제한 |

## 8. 리포트 표현 차이

개인 MVP:

```text
제출 전 본인이 직접 확인할 보완 항목
```

B2B:

```text
조직 기준에 맞춰 추가 확인이 필요한 항목
```

B2B에서도 작성자를 단정하지 않는 정책은 유지한다.

## 9. B2B 금지 카피

B2B 화면과 영업 자료에서는 아래 방향의 표현을 피한다.

| 금지 방향 | 대체 방향 |
|---|---|
| 구성원을 감시하는 도구처럼 보이는 표현 | 제출 전 확인 흐름을 표준화하는 도구 |
| 작성 의도를 판정하는 표현 | 문서 내 확인 신호를 정리하는 표현 |
| 부정행위 적발처럼 보이는 표현 | 근거 보완과 정책 체크를 돕는 표현 |
| 리포트만으로 평가가 끝나는 표현 | 사람의 검토를 돕는 참고 자료 |

## 10. 사람 검토와 이의 제기 흐름

교육, 채용, HR 맥락에서는 보고서만으로 불이익을 주지 않는 절차가 필요하다.

```text
리포트 생성
→ 제출자 본인 확인
→ 검토자 코멘트
→ 제출자 소명 또는 수정본 제출
→ 사람 검토
→ 최종 판단은 별도 기준으로 진행
```

필수 원칙:

- 리포트는 보완 참고 자료로만 사용한다.
- 최종 판단은 사람 검토와 기존 평가 기준을 따른다.
- 제출자는 리포트 해석에 대한 설명 또는 수정 기회를 가져야 한다.
- 리포트와 원문 접근 권한은 역할별로 제한한다.

## 11. 구현 순서 제안

| 순서 | 작업 |
|---:|---|
| 1 | 개인 MVP 내비게이션 유지 |
| 2 | `/org` 일반 메뉴 숨김 유지 |
| 3 | B2B 라우트는 별도 진입 링크로만 접근 |
| 4 | B2B 화면에 프로토타입/데모 고지 추가 |
| 5 | 역할/권한 매트릭스 문서 확정 |
| 6 | 계정 및 조직 저장 정책 승인 후 구현 |

B2B 계정, 조직 저장, 원문 열람, 공유 링크는 Sprint 3 설계 확정 이후 별도 Sprint에서 다룬다.

## 12. 승인 전 금지 사항

- B2B 메뉴를 일반 사용자에게 노출
- 조직 관리자 원문 열람 구현
- 계정/조직 저장 기능 구현
- 실제 사용자 파일럿처럼 B2B 데모 소개
- 작성자 판정/감시 도구처럼 표현
